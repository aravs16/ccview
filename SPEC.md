# ccview вҖ” Product Spec

> A companion app for Claude Code that gives AI output a visual layer.
> CC writes JSON files. ccview renders them. That's it.

---

## Problem

Claude Code produces rich structured output вҖ” financial reports, research, code analysis, test results, deployment status, data tables. But the terminal is a terrible medium for consuming structured information. Every CC user eventually hits the wall: great data, can't *see* it.

Users today work around this by:
- Asking CC to write markdown files, then opening them manually
- Building custom apps (Next.js, Tauri) for each use case
- Scrolling back through terminal history looking for that table CC printed 20 minutes ago
- Copy-pasting output into Notion or Google Sheets

All of these are friction. The output is ephemeral and hard to find.

## Solution

**ccview** is a local TUI app that watches a directory for JSON files and renders them as visual pages. Claude Code writes files. ccview shows them. No servers, no APIs, no MCP, no config.

```
Claude Code (writes JSON) вҶ’ ~/.ccview/pages/ вҶ’ ccview (renders)
```

### How CC knows the format

ccview ships with a Claude Code skill file at `~/.claude/skills/view/SKILL.md`. This file contains the complete block schema. When CC reads it, it knows exactly how to produce ccview-compatible output.

Any existing skill can opt in by adding one line:

```
If ~/.ccview/pages/ exists, also publish results there using the ccview format.
```

CC checks if the directory exists. If ccview is installed вҶ’ writes the file alongside normal output. If not installed вҶ’ nothing happens. Zero coupling.

### Installation

```bash
brew install ccview     # macOS
cargo install ccview    # cross-platform
npx ccview              # try without installing
```

Installation does two things:
1. Places the `ccview` binary in PATH
2. Creates `~/.claude/skills/view/SKILL.md` (the protocol spec for CC)
3. Creates `~/.ccview/pages/` directory

### Usage

```bash
ccview                  # launch TUI, watches ~/.ccview/pages/
ccview --web            # launch web UI on localhost:3141 instead
ccview --export <page>  # export a page as standalone HTML
ccview --clean          # delete pages older than 7 days
ccview --clear          # delete all pages
```

## The Protocol

### File location

```
~/.ccview/pages/{id}.json
```

- `{id}` is any string вҖ” typically kebab-case (`financial-report-march-2026`)
- CC picks the id. If it writes the same id again, it's an update (overwrite)
- The directory is flat. No nesting.

### Page schema

```json
{
  "title": "Financial Report вҖ” March 2026",
  "subtitle": "Generated by /financial-analyst",
  "icon": "bar-chart",
  "created": "2026-04-15T10:30:00Z",
  "updated": "2026-04-15T10:30:00Z",
  "pinned": false,
  "tags": ["finance", "monthly"],
  "ttl": "7d",
  "blocks": [ ... ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | yes | Page title shown in sidebar |
| `subtitle` | string | no | Secondary text under title |
| `icon` | string | no | Lucide icon name (default: `file-text`) |
| `created` | ISO 8601 | yes | When the page was first created |
| `updated` | ISO 8601 | yes | Last modification time |
| `pinned` | boolean | no | Pinned pages persist, shown at top (default: false) |
| `tags` | string[] | no | For filtering in sidebar |
| `ttl` | string | no | Auto-delete after duration: `1h`, `1d`, `7d`, `30d`, `forever` (default: `7d`) |
| `blocks` | Block[] | yes | Ordered list of content blocks |

### Block types

Every block has a `type` field and type-specific data. ccview renders blocks top-to-bottom in the order they appear.

---

#### `metrics` вҖ” Big number cards

The dashboard staple. Shows key values in a horizontal row.

```json
{
  "type": "metrics",
  "items": [
    {
      "label": "Income",
      "value": "$35,596",
      "change": "+12.3%",
      "sentiment": "positive"
    },
    {
      "label": "Expenses",
      "value": "$28,061",
      "change": "+3.0%",
      "sentiment": "negative"
    },
    {
      "label": "Net",
      "value": "+$7,534",
      "sentiment": "positive"
    }
  ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `items[].label` | string | yes | Metric name |
| `items[].value` | string | yes | Display value (pre-formatted) |
| `items[].change` | string | no | Delta text (e.g. "+5%", "-$200") |
| `items[].sentiment` | `positive \| negative \| neutral` | no | Colors the value green/red/gray |

**TUI rendering:** Horizontal boxes with label above, value large, change below.
**Web rendering:** Card grid with optional background tint based on sentiment.

---

#### `table` вҖ” Sortable data table

```json
{
  "type": "table",
  "title": "Top Merchants",
  "sortable": true,
  "columns": [
    { "key": "merchant", "label": "Merchant" },
    { "key": "amount", "label": "Amount", "align": "right" },
    { "key": "category", "label": "Category" }
  ],
  "rows": [
    { "merchant": "Truist Mortgage", "amount": "$6,800", "category": "Loan" },
    { "merchant": "AWS", "amount": "$1,369", "category": "Tech" }
  ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | no | Section title above table |
| `sortable` | boolean | no | Enable column sort (default: false) |
| `filterable` | boolean | no | Show filter input (default: false) |
| `columns` | Column[] | yes | Column definitions |
| `columns[].key` | string | yes | Key in row objects |
| `columns[].label` | string | yes | Display header |
| `columns[].align` | `left \| right \| center` | no | Text alignment (default: left) |
| `rows` | object[] | yes | Row data keyed by column keys |

Columns can also be simplified as a string array: `"columns": ["Merchant", "Amount"]` вҖ” keys are auto-derived as lowercase.

**TUI rendering:** Ratatui table widget with scrolling, optional sort on keypress.
**Web rendering:** HTML table with sticky headers, click-to-sort.

---

#### `chart` вҖ” Visual data charts

```json
{
  "type": "chart",
  "title": "Monthly Spend Trend",
  "chart": "bar",
  "data": [
    { "label": "Jan", "value": 26784 },
    { "label": "Feb", "value": 25026 },
    { "label": "Mar", "value": 28061 }
  ]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | string | no | Chart title |
| `chart` | `bar \| line \| sparkline \| pie \| hbar` | yes | Chart type |
| `data` | DataPoint[] | yes | Chart data |
| `data[].label` | string | yes | X-axis label or pie slice name |
| `data[].value` | number | yes | Y-axis value |
| `data[].color` | string | no | Override color for this point |

For multi-series charts:

```json
{
  "type": "chart",
  "chart": "line",
  "series": [
    { "name": "Income", "color": "green", "data": [{"label": "Jan", "value": 15000}, ...] },
    { "name": "Expenses", "color": "red", "data": [{"label": "Jan", "value": 26000}, ...] }
  ]
}
```

**TUI rendering:** Ratatui bar chart or sparkline widget (braille dots for line charts).
**Web rendering:** SVG charts (lightweight, no D3 dependency).

|-------|------|----------|-------------|

**Navigation:**
- `вҶ‘/вҶ“` or `j/k` вҖ” navigate pages in sidebar
- `Enter` вҖ” open page
- `Tab` вҖ” switch focus between sidebar and content
- Scroll content with arrow keys or mouse wheel

**Tech:** Rust + Ratatui + notify (file watcher) + syntect (code highlighting)

### Web Mode (`--web`)

Same data, richer rendering. Launches a local HTTP server serving a single-page app.

- Charts rendered as actual SVG (not ASCII)
- Tables with real sorting/filtering
- Markdown with full formatting
- Click-to-copy on code blocks
- Responsive layout

**Tech:** Rust + axum + embedded static HTML/CSS/JS (no build step, no npm)

The web UI is embedded in the same binary. No separate install.

### Export Mode (`--export`)

```bash
ccview --export financial-report-march-2026 > report.html
```

Produces a self-contained HTML file with inline CSS/JS. Can be shared, emailed, or opened anywhere.

## Page Lifecycle

### Creation
CC writes a JSON file to `~/.ccview/pages/`. ccview detects it via file system watcher (fsnotify). Page appears in sidebar immediately.

### Updates
CC overwrites the same file. ccview re-renders. This enables live-updating pages вҖ” a long-running skill can write incremental updates.

### TTL & Cleanup
Each page has a `ttl` field (default: `7d`). ccview auto-cleans expired pages on startup and periodically. Pinned pages ignore TTL.

```bash
ccview --clean          # manual cleanup of expired pages
ccview --clear          # delete everything
```

### Pinning
Users can pin pages from the TUI (`p` key) or web UI. Pinned pages:
- Show at the top of the sidebar
- Never auto-expire
- Survive `--clean`

Pinning is stored in a local `~/.ccview/state.json` file (not in the page JSON, so CC overwrites don't lose pin state).

## The Skill File

Installed at `~/.claude/skills/view/SKILL.md`:

```markdown
---
name: view
description: Publish visual output to the ccview companion app
allowed-tools:
  - "Bash"
---

# ccview вҖ” Visual Output

When you want to display results visually вҖ” tables, charts,
metrics, reports вҖ” write a JSON file to `~/.ccview/pages/`.

## Quick Start

\`\`\`bash
cat > ~/.ccview/pages/my-page.json << 'EOF'
{
  "title": "Page Title",
  "created": "{{ISO_DATE}}",
  "updated": "{{ISO_DATE}}",
  "blocks": [
    { "type": "markdown", "content": "# Hello World" }
  ]
}
EOF
\`\`\`

## Block Types

[... full schema for all 12 block types ...]

## Guidelines

- Use kebab-case for page IDs (filename without .json)
- Always set created/updated to current ISO 8601
- Keep pages focused вҖ” one topic per page
- Use metrics blocks for KPIs, tables for data, charts for trends
- Use callout blocks to highlight important findings
- Set ttl: "forever" for dashboards, "1d" for ephemeral reports
```

This is the entire integration surface. No SDK. No API. CC reads this skill, knows the format, writes files.

## How Existing Skills Integrate

A skill like `/financial-analyst` adds this to the end of its SKILL.md:

```markdown
## Visual Output

After generating the markdown report, also publish to ccview
if the directory `~/.ccview/pages/` exists.

Write a JSON file to `~/.ccview/pages/financial-{month}.json`
using the ccview block format:
- metrics block: income, expenses, net
- table block: top merchants, category breakdown
- chart block: monthly trend (bar chart)
- callout block: any red flags
- timeline block: upcoming events
- list block: savings action items
```

That's it. The skill author describes what blocks to produce. CC figures out the mapping from its analysis output to ccview blocks.

## Distribution

### Package Managers

```bash
brew install ccview              # Homebrew (macOS/Linux)
cargo install ccview             # Rust ecosystem
npx ccview                       # Try without installing (Node)
scoop install ccview             # Windows
```

### What the installer does

1. Places `ccview` binary in PATH
2. Creates `~/.ccview/pages/` directory
3. Copies `view/SKILL.md` to `~/.claude/skills/view/SKILL.md` (if ~/.claude exists)
4. Prints: "ccview installed. Run `ccview` to start. Your CC skills can now push visual output."

### Uninstall

```bash
brew uninstall ccview
rm -rf ~/.ccview ~/.claude/skills/view
```

Clean. No orphaned processes, no background services.

## Non-Goals (v1)

- **Not a dashboard builder.** Users don't design pages. CC writes them.
- **Not a notebook.** No editing, no execution. Read-only viewer.
- **Not a collaboration tool.** Local only. No sharing, no auth, no cloud.
- **Not an MCP server.** File system is the protocol. Simpler, dumber, better.
- **No real-time data.** Pages update when CC writes them, not on a schedule. (Skills can implement their own polling.)

---

## Future (v2+)

- **Page templates** вҖ” predefined layouts for common skill outputs (finance report, PR review, test results, deploy status)
- **Interactive blocks** вҖ” checkboxes that write back to the JSON (for action items)
- **Page linking** вҖ” blocks that reference other pages
- **Search** вҖ” full-text search across all pages
- **Themes** вҖ” dark mode, custom color schemes
- **Team sharing** вҖ” `ccview share <page>` generates a shareable link (requires cloud service)
- **Webhooks** вҖ” notify external services when a page is created/updated
- **Plugin block types** вҖ” custom renderers for domain-specific visualizations
